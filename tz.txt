# Тестове завдання: Mini-додаток «Warehouse Catalog»

Адмін-панель для управління каталогом товарів складу. Завдання покриває весь робочий стек

---

## 1. Технологічний стек (обовʼязковий)

| Призначення | Бібліотека |
|---|---|
| Фреймворк | Next.js 15 (App Router) + React 19 |
| Мова | TypeScript |
| Стилізація | Tailwind CSS v4, `tailwind-merge` (утиліта `cn()`) |
| Server state | `@tanstack/react-query` v5 |
| HTTP | `axios` (з єдиним інстансом + інтерсептори) |
| Client state | `valtio` |
| Форми / селекти | `react-select`, `react-select-async-paginate` |
| Дата/час | `flatpickr` |
| Завантаження файлів | `react-dropzone` |
| Штрихкод | `react-barcode` |
| Рейтинг | `react-simple-star-rating` |
| Tooltip / popover | `@popperjs/core` |
| Debounce | `use-debounce` |
| SVG як React-компоненти | `@svgr/webpack` |
| Код-стайл | ESLint + Prettier (+ `prettier-plugin-tailwindcss`) |

---

## 2. Опис проекту

Тобі треба зробити адмінку каталогу товарів. Дані беруться з мок-API (інструкція нижче).

Сторінки:

1. `/products` — список товарів з фільтрами, пошуком та пагінацією.
2. `/products/new` — створення товару.
3. `/products/[id]` — перегляд/редагування товару.
4. `/suppliers` — список постачальників (мінімальна таблиця, без редагування).

---

## 3. Функціональні вимоги

### 3.1. Сторінка `/products` (список)

- Таблиця: `Зображення (мініатюра) | Назва | SKU/штрихкод | Категорія | Постачальник | Ціна | Рейтинг | Дата надходження | Дії`.
- Дані тягнути через `useQuery` (React Query). Ключі мають включати фільтри (`['products', { search, categoryId, dateRange, page }]`).
- **Пошук за назвою** — `<input>` з debounce 400 мс через `use-debounce`. Не змінювати URL миттєво — тільки після дебаунсу.
- **Фільтр за категорією** — `react-select-async-paginate` (підвантаження сторінками з API).
- **Фільтр за діапазоном дат надходження** — `flatpickr` у режимі range.
- **Сортування** за ціною та датою (клік по заголовку колонки).
- **Пагінація** серверна (page/limit).
- Стан фільтрів і сортування зберігати у **Valtio store** (а не у `useState`), щоб його можна було читати з будь-якого компонента сторінки. Синхронізувати з query-параметрами URL (зміна фільтра → змінюється URL, перезавантаження сторінки відновлює стан).
- Скелетон-лоадери під час фетчу.

### 3.2. Сторінка `/products/new` і `/products/[id]` (форма)

Форма має поля:

- Назва (текст, обовʼязкове, мін. 2 символи).
- Опис (textarea).
- Ціна (число > 0).
- SKU (генерується автоматично, якщо не введено вручну) — поруч з полем рендериться превʼю штрихкоду через `react-barcode`. При зміні SKU превʼю оновлюється.
- Категорія — `react-select-async-paginate`.
- Постачальник — звичайний `react-select` (постачальників ≤ 50, тягнеться один раз).
- Дата надходження — `flatpickr` (single).
- Зображення товару — `react-dropzone`, до 5 файлів, drag&drop + клік. Превʼю мініатюр з можливістю видалити окреме фото. Валідація: тільки `image/*`, кожен файл ≤ 2 MB.
- Внутрішній рейтинг якості — `react-simple-star-rating` (1–5).
- Активний/неактивний — звичайний чекбокс.

Додатково:

- На іконці «?» біля деяких полів — **tooltip** з підказкою. Реалізувати власний компонент `<Tooltip>` поверх `@popperjs/core` (без сторонніх tooltip-обгорток).
- Збереження через `useMutation`. Після успіху — інвалідація `['products']` і `['product', id]`, редірект на `/products` з тостом.
- При завантаженні `/products/[id]` — `useQuery` витягує товар і `reset` форми його значеннями.
- Валідація форми — на твій вибір (`react-hook-form + zod` чи свій варіант), але має бути inline-помилки під полями.

### 3.3. Сторінка `/suppliers`

- Проста таблиця з даними постачальників (`useQuery`). Призначена, щоб показати, що ти вмієш переюзовувати компоненти таблиці/пагінації.

### 3.4. Загальне

- **Layout** з бічним меню (Products / Suppliers) і хедером. Іконки в меню — SVG-компоненти, імпортовані через `@svgr/webpack` (`import IconBox from '@/icons/box.svg'`).
- **Axios instance** у `src/lib/api.ts` з `baseURL` з `process.env.NEXT_PUBLIC_API_URL`, інтерсептором на помилки (показувати тост при 5xx).
- Глобальний `<QueryClientProvider>` з налаштуваннями за замовчуванням (`staleTime`, `retry`, `refetchOnWindowFocus: false`).
- Утиліта `cn(...classes)` на базі `tailwind-merge` для злиття класів.
- Тости / нотифікації — простий власний компонент, керований через Valtio store (`toastsStore`).

## 4. Mock API

Використати `json-server` (або `MockAPI.io`). У репозиторії додати `db.json` із сидами:

```
/products?_page=1&_limit=20&q=...&categoryId=...&_sort=price&_order=asc
/products/:id
/categories?_page=1&_limit=20
/suppliers
```

Скрипт `npm run mock` має піднімати mock-сервер на `http://localhost:4000`.

## 6. Що ми оцінюємо

1. **Архітектура та структура файлів** — feature-based, відсутність хаосу.
2. **TypeScript** — без `any`, нормальні типи на API-відповіді, generic-хуки де доречно.
3. **React Query** — правильні ключі, інвалідація після мутацій, відсутність дублювання запитів, винесення хуків (`useProductsQuery`, `useProductMutation`).
4. **Valtio** — розуміння різниці між server state (RQ) і UI state (Valtio). Не дублювати дані, які вже є у RQ.
5. **Форми** — валідація, дизейбл сабміту, обробка помилок API.
6. **Доступність** — `<label>`, `aria-*`, фокус-стани, навігація з клавіатури.
7. **UX-дрібниці** — скелетони, оптимістичні апдейти (бонус), стани empty/error.
8. **Чистий код** — ESLint без помилок, Prettier, осмислені імена, відсутність мертвого коду.
9. **README** — як запустити (`npm i`, `.env.example`, `npm run mock`, `npm run dev`), скріншоти.

## 7. Бонус (необовʼязково, але плюс)

- Optimistic update при зміні `active`-чекбоксу у списку.
- Bulk-actions: масове видалення/деактивація через чекбокси у таблиці.
- Error boundary на рівні сторінки.
- Юніт-тести для одного хука й однієї утиліти (Vitest або Jest).
- Експорт списку товарів у CSV.

---

## 8. Чого робити НЕ потрібно

- Авторизація / логін.
- Реальний бекенд.
- SSR/RSC-фетчинг (App Router використовуй, але всі дані тягни на клієнті через React Query — це робочий патерн на проекті).
- Дизайн «під Figma» — головне, щоб виглядало охайно і консистентно.
